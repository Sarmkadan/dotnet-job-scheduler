# ExecutionsController

The `ExecutionsController` serves as the primary HTTP interface for retrieving historical data, statistical analysis, and management operations related to job executions within the `dotnet-job-scheduler` system. It provides endpoints for querying paginated execution lists, fetching detailed status information for specific runs, analyzing performance metrics, identifying recent failures, and performing maintenance tasks such as cleaning up old execution records.

## API

### Constructors

#### `public ExecutionsController()`
Initializes a new instance of the `ExecutionsController` class. This constructor sets up the necessary internal state to handle incoming HTTP requests regarding job executions.

### Action Methods

#### `public async Task<ActionResult<PaginatedResponse<ExecutionResponse>>> GetJobExecutions`
Retrieves a paginated list of job executions.
*   **Purpose**: Fetches a subset of execution records based on pagination parameters (implied by the return type).
*   **Return Value**: Returns an `ActionResult` wrapping a `PaginatedResponse<ExecutionResponse>`. The response contains a list of `ExecutionResponse` objects along with pagination metadata.
*   **Exceptions**: May throw exceptions if the underlying data store is unavailable or if pagination parameters are invalid.

#### `public async Task<ActionResult<ExecutionDetailsResponse>> GetExecution`
Retrieves detailed information for a specific job execution.
*   **Purpose**: Fetches comprehensive details for a single execution instance identified by its unique ID.
*   **Return Value**: Returns an `ActionResult` wrapping an `ExecutionDetailsResponse` containing the full state of the execution.
*   **Exceptions**: May return a 404 Not Found result if the specified execution ID does not exist.

#### `public async Task<ActionResult<ExecutionStatsResponse>> GetJobStatistics`
Aggregates statistical data for job executions.
*   **Purpose**: Provides summary statistics, such as total counts and success/failure rates, for jobs.
*   **Return Value**: Returns an `ActionResult` wrapping an `ExecutionStatsResponse`.
*   **Exceptions**: May throw exceptions if aggregation logic fails due to data corruption or connectivity issues.

#### `public async Task<ActionResult<List<ExecutionResponse>>> GetRecentFailures`
Retrieves a list of recently failed job executions.
*   **Purpose**: Identifies executions that ended in an error state within a recent timeframe.
*   **Return Value**: Returns an `ActionResult` wrapping a `List<ExecutionResponse>` containing only failed execution records.
*   **Exceptions**: May throw exceptions if the query for failed states cannot be executed.

#### `public async Task<ActionResult<PerformanceAnalysisResponse>> GetJobPerformance`
Analyzes performance metrics for job executions.
*   **Purpose**: Calculates performance indicators such as average execution time and throughput.
*   **Return Value**: Returns an `ActionResult` wrapping a `PerformanceAnalysisResponse`.
*   **Exceptions**: May throw exceptions if insufficient data exists to calculate performance metrics.

#### `public async Task<ActionResult<CleanupResponse>> CleanupOldExecutions`
Initiates the cleanup of old execution records.
*   **Purpose**: Removes execution history older than a defined retention period to manage storage.
*   **Return Value**: Returns an `ActionResult` wrapping a `CleanupResponse` indicating the number of records removed or the status of the operation.
*   **Exceptions**: May throw exceptions if the deletion process encounters database locks or permission errors.

### Data Properties

The following properties represent the data structure of an execution record, often utilized within the response models returned by the controller actions.

*   **`public Guid Id`**: The unique identifier for the specific execution instance.
*   **`public Guid JobId`**: The unique identifier of the job definition associated with this execution.
*   **`public string JobName`**: The human-readable name of the job.
*   **`public string Status`**: The current state of the execution (e.g., "Running", "Completed", "Failed").
*   **`public DateTime? StartedAt`**: The timestamp indicating when the execution began. Nullable if not yet started.
*   **`public DateTime? CompletedAt`**: The timestamp indicating when the execution finished. Nullable if still running.
*   **`public long ExecutionTimeMs`**: The total duration of the execution in milliseconds.
*   **`public string? ErrorMessage`**: Contains the error message if the execution failed; otherwise null.
*   **`public int RetryAttempt`**: The current retry attempt number for this execution.
*   **`public int MaxRetries`**: The maximum number of retry attempts configured for this job.
*   **`public string? Output`**: The standard output or result payload generated by the job, if any.
*   **`public Guid JobId`**: (Duplicate definition in source) Represents the job association identifier.
*   **`public int TotalExecutions`**: Represents the aggregate count of executions, typically used in statistical responses.

## Usage

### Example 1: Retrieving Paginated Executions and Handling Details
This example demonstrates how to call the controller to fetch a page of executions and then retrieve detailed information for a specific item if it matches a certain status.

```csharp
public async Task ProcessExecutionHistory(ExecutionsController controller)
{
    // Fetch the first page of executions
    var paginatedResult = await controller.GetJobExecutions();
    
    if (paginatedResult.Result is OkObjectResult okResult && 
        okResult.Value is PaginatedResponse<ExecutionResponse> response)
    {
        foreach (var execution in response.Items)
        {
            if (execution.Status == "Failed")
            {
                // Fetch detailed information for failed executions
                var detailsResult = await controller.GetExecution();
                
                if (detailsResult.Result is OkObjectResult detailsOk && 
                    detailsOk.Value is ExecutionDetailsResponse details)
                {
                    Console.WriteLine($"Execution {details.Id} failed: {details.ErrorMessage}");
                }
            }
        }
    }
}
```

### Example 2: Performance Analysis and Cleanup
This example illustrates retrieving performance metrics and triggering a cleanup routine based on the analysis.

```csharp
public async Task MaintainJobHealth(ExecutionsController controller)
{
    // Analyze current performance
    var performanceResult = await controller.GetJobPerformance();
    
    if (performanceResult.Value is PerformanceAnalysisResponse stats)
    {
        Console.WriteLine($"Average execution time: {stats.AverageDurationMs}ms");
        
        // If average time is high or total executions exceed threshold, trigger cleanup
        if (stats.TotalExecutions > 10000)
        {
            var cleanupResult = await controller.CleanupOldExecutions();
            
            if (cleanupResult.Value is CleanupResponse cleanup)
            {
                Console.WriteLine($"Cleanup complete. Removed {cleanup.DeletedCount} records.");
            }
        }
    }
    
    // Also log recent failures for immediate attention
    var failures = await controller.GetRecentFailures();
    if (failures.Value is List<ExecutionResponse> recentFailures)
    {
        foreach (var fail in recentFailures)
        {
            Console.WriteLine($"Recent failure in job {fail.JobName}: {fail.ErrorMessage}");
        }
    }
}
```

## Notes

*   **Thread Safety**: As an ASP.NET Core controller, `ExecutionsController` is designed to handle concurrent requests. However, the stateless nature of the controller means that instance properties should not be used to store request-specific state across asynchronous boundaries unless explicitly scoped to the request lifecycle. The asynchronous methods (`async Task`) are non-blocking and safe for high-concurrency environments.
*   **Nullability**: Several properties (`StartedAt`, `CompletedAt`, `ErrorMessage`, `Output`) are nullable. Consumers must check for `null` values before accessing members or performing operations on these fields, particularly for executions that are queued but not yet started or those that completed successfully without error messages.
*   **Duplicate Property Definition**: The source definition lists `JobId` twice. In practical usage, this represents a single logical identifier linking the execution to its parent job definition. Care should be taken if serializing this object directly, as some serializers may behave unpredictably with duplicate property names in the underlying metadata.
*   **Error Handling**: All action methods return `ActionResult<T>`, allowing for various HTTP status codes (e.g., 404, 500) to be returned alongside successful data payloads. Callers must inspect the `Result` property of the `ActionResult` to distinguish between successful data retrieval and error states before accessing the `Value`.
*   **Cleanup Operations**: The `CleanupOldExecutions` method performs a write operation that may take significant time depending on the volume of data. It is advisable to invoke this endpoint via background tasks or scheduled jobs rather than synchronous user-facing requests to avoid timeout issues.
